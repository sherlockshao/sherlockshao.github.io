CRM Report基本设置与介绍

  1. 新建Report Solution，引用 Report Related Category 和Report 这两个系统Entity。
  2. 在Solution的Components中选中Reports，new一个看看有哪些设置。
    Report Type
    Report Wizard Report：在界面一步步新建report，可以设置简单的Report,不够灵活。
    Existing File:添加开发好的report文件，项目中使用这种方法。
    Link to Webpage：可以跳转到特定的url。
    Parent Report
    如果是subreport，这一项需要lookup到主report上，添加existing file时会自动关联上。
    Categories
    设置Report的Category
    Related Record Types
    Report指定显示在哪个Entity下。
    Display In
    Reports area:可以再CRM report的view中显示。
    Forms for related record types:在Form中的Run Report Button下显示。
    Lists for related record types:在View中的Run Report Button下显示。
  3. Report Related Category Entity的Fields中找到 categorycode，发现里面有4个系统自带的value，添加项目实际需求的Report Category，比如”LPA Reports”，”Deputy Reports”等。
  4. CRM有很多build-in report,这些report对于定制的系统来说完全是多余的，需要隐藏。可通过不同的view来控制隐藏。
    Report Entity的Views中新建view如”Available Reports”，添加Filter设置Report Category为#3中自定义的Category，可根据需求设置Visibility Equals Reports area。
  5. 不要修改Reports View下的Default Public View
    如果deactive report下原有的view，在solution导入新环境后不会生效，需要手动再去deactive。
    如果修改了report view下的Default Public View，在solution导入新环境之后系统默认的Available Reports仍然是Default Public View，此时就会出现两个Default Public View，有意思的是实际的Default Public View为排在后面的那个。而修改default view只会把原先排在最上面的default view置成普通public view。而default view又deactive不了。在重新设置default view时发现问题很多。这段说的有点绕，不需要理解。开发环境的解决方案是重导unmanned solution，把view的Default Public View值由1改成0。保证只有系统默认的Default View生效。导入solution再设置成想要的view为Default View。

Report文件基本结构介绍

用Sql Server Data Tool新建或编译一个report文件，看下文件结构。

布局

从布局角度看，Report包含页眉，页脚和正文部分，页眉页脚为每一页重复的部分，需要注意的是subreport中的页眉页脚并不起作用。report导出成word，页眉页脚的背景色不起作用，导出成word还有不少其他问题，导出成其他类型文件的限制也可以在那里找到。

Report Properties

  • Page Setup：可以设置report导出后每页的大小和页边距。report中所有的距离单位只有in, mm, cm, pt, pc这5种，Pager Size可以直接选择A4纸的大小。
  • Code：可以添加自定义的代码，但目前只能添加VB代码，暂不支持C#,在使用时只要调用Code.方法名 就可以使用，之后可能会贴一些写的方法。
  • References：添加引用，可以是上一篇提到的Microsoft.Crm.Reporting.RdlHelper.dll，也可以是自己build的assembly，不过自己build的文件不能导入CRM，所以我没使用这种方法。
  • Variable：添加变量，这个没用过，了解不多。倒是在Code中添加过全局变量。

Report Data

  • data sources：connection string连接开发环境就行，不用改，当report导入到crm之后，report的data source会变成CRM Report Service中的data source，也就是CRM自身。
  • Build-in Fields:Report会自带一些fields,常用的有Page Number(当前页号)，Total Pages(总页号)，另外有个Field是User ID,虽然这个没用到过，但是这个field调用了Sql Server的一个存储过程来获取usre id。之前遇到过run report出错是因为report Service的user对这个存储过程没有权限，通过log发现的，赋下权限就能解决。
  • Images：以字节数组的形式存在xml文件的节点下。在开发过程中遇到过jpg后缀的图片比png后缀的图片清楚。
  • Parameters：可以定义参数DatePattern，默认值为”dd/mm/yyyy”，在实现显示使用系统方法Format(Fields!xxxxxx.Value.ToLocalTime,Parameters!CRM_DatePattern.Value)就可以准换成统一格式的时间，另外参数设置成hidden，否则每次preview都需要手动填值。
    项目中有通过代码生成Report的需求，为了可以定位到具体某一个Record，对于所有Report我添加名为CRM_EntityPrimaryKey的参数，来接收代码传递来的具体Entity的PrimaryKey。
  • DataSets：在这写Fetch语句，report中的data都是从这查询的。在查询语句最外层的Entity中需要添加Prefilter来保证run report时查询的是当前record的数据。 <entity name="xx" enableprefiltering="1" prefilterparametername="CRM_Filterexxxx">
    save之后会发现多处一个parameter，name是 “CRM_Filterexxxx”，defaultvale为 <fetch version="1.0" output-format="xml-platform" mapping="logical" distinct="false"><entity name="xx"><all-attributes/></entity></fetch>
    当然也可以预先定义好这个参数，然后在dataset中直接拿来用。在crm中run这个report时会发现report的Filter条件里会有查询当前record的设置。
    通过代码调用怎么来控制取哪个record下的数据呢？我的思路是这样的，利用刚才提到的 CRM_EntityPrimaryKey 对DataSet设置Filter条件为： 当CRM_EntityPrimaryKey有值时，Entity主键的值等于CRM_EntityPrimaryKey的值，否则Entity主键的值等于自身，这样代码调用时就可以取指定record的数据了。
    还有一种情况下会根据某一次查询的值去查询另一个record的数据，这时可以自定义一个参数@CRM_ID，默认值选dataset中查询的某个filed的值，然后再建一个dataset，Fetch语句里添加Filter条件
    <condition attribute="primarykey" operator="eq" value="@CRM_ID" />
    要保证两个dataset的先后顺序，否则报错。

关于通过代码调用有一点更新，按照之前的方式在生产环境中存在严重效率问题。对DataSet设置Filter，相当于先把所有数据查询出来，再在内存中做Filter，通过code调用ssrs时，默认超时时间是100s,会导致timeout。解决办法是复制一份report文件用于代码调用，在fetch xml中添加过滤条件，经测试，1w多条数据，效率可提升100多倍。之前考虑过可能出现这个问题，但通过改fetch xml，没找到办法可以和crm中run report的文件公用，就是需要维护两份几乎一样的report，并且测试过程中没发现明显的效率问题，所以一直也没太在意这个问题，直到问题的出现。
看来以后还得多家小心，Report一定要考虑数据量的问题。
